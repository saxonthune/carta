# The .carta/ Workspace Format

A `.carta/` workspace is a directory of Markdown files that holds a software project's specifications. It lives at the root of a repository, alongside the source code it describes.

The format is designed for two readers: humans browsing in an editor, and AI agents searching programmatically. Plain Markdown keeps it readable. Structured frontmatter and a generated index keep it searchable.

## Structure

A workspace is organized into numbered **titles**:

```
.carta/
  MANIFEST.md              # generated index (don't edit by hand)
  00-codex/                # how to use these docs
  01-context/              # mission, principles, glossary
  02-system/               # architecture, decisions, interfaces
  03-product/              # features, use cases, workflows
  04-operations/           # dev setup, testing, deployment
```

Title 00 (codex) is universal — it ships with every workspace and documents the workspace itself. The rest are up to you.

Within titles, documents are numbered Markdown files with kebab-case slugs:

```
01-context/
  00-index.md
  01-mission.md
  02-principles.md
  03-glossary.md
```

Directories nest arbitrarily deep. When a document outgrows its file, it becomes a directory with its own `00-index.md` and children.

Every directory should have a `00-index.md` that explains what the directory contains and why it's organized the way it is. This is the human entry point — what you read when you open a folder and want to orient yourself.

```
01-context/00-index.md
```

```markdown
---
title: Context
status: active
summary: Mission, principles, and shared vocabulary
tags: [index, context]
deps: []
---

# Context

Why this project exists, what it values, and the terms we use.

- **01-mission** — the one-sentence reason for the project
- **02-principles** — design values that guide decisions
- **03-glossary** — canonical definitions for domain terms
```

## Documents

Every document starts with YAML frontmatter:

```yaml
---
title: Design Principles
status: active
summary: One-line description for the MANIFEST
tags: [principles, design]
deps: [doc01.01]
---
```

`status` is one of `draft`, `active`, or `deprecated`. `deps` lists documents that should be checked when this one changes. `tags` feed the MANIFEST's tag index for retrieval.

The body is plain Markdown. When a document needs referenceable subsections, mark them with `## #secNN` headings:

```markdown
## #sec01 Inverse Derivability

Content here.

## #sec02 Arrangement Agnosticism

Content here.
```

Most documents don't need sections. They appear when the work demands atomic references into a document.

## Cross-References

Documents reference each other with `docXX.YY.ZZ` syntax, where each two-digit segment maps to a directory or file prefix:

```
doc01.03.02
 │    │   └─ file 02-principles.md
 │    └───── subdir 03-product-strategy/
 └────────── title 01-context/
```

References survive renames and moves — the `carta` CLI rewrites them automatically when the workspace structure changes.

## MANIFEST.md

`MANIFEST.md` is a generated index at the workspace root. It contains a table of every document (ref, file path, summary, tags, dependencies) and a tag index mapping keywords to document refs.

AI agents read the MANIFEST first to locate relevant documents without scanning every file. It is regenerated by `carta regenerate` and should not be edited by hand.

## The carta CLI

The `carta` CLI handles workspace operations: creating, moving, deleting, renaming, and reorganizing documents. It manages numbering, gap-closing, cross-reference rewriting, and MANIFEST regeneration.

```bash
pip install carta-cli
carta init
```

See the [CLI README](README.md) for usage.
